# Atlas Glossary

## Purpose

This document defines common terms used throughout Atlas.

The goal is to ensure a shared understanding between:

- Humans

- Agents

- Documentation

- Architecture

- Implementation

All project documents should use these definitions consistently.

---

## Project

A Project is the primary entity managed by Atlas.

A Project represents a software system that Atlas is responsible for understanding and documenting.

A Project contains metadata and references one or more information sources.

Examples:

- Payment Service

- E-Commerce Platform

- Internal Developer Portal

### Version 0.1

In Version 0.1, a Project references exactly one Repository.

---

## Repository

A source code repository connected to Atlas.

A Repository is the primary source of information for Atlas analysis and knowledge generation.

Atlas reads repositories to extract facts, generate summaries, and build project understanding.

Examples:

- GitHub Repository

- GitLab Repository

- Bitbucket Repository

### Relationship to Project

A Repository and a Project are not the same concept.

- Repository = Source of information

- Project = Entity managed by Atlas

In Version 0.1:

```text
Project
    |
    +---- One Repository
```

Each Project references exactly one Repository.

### Future Considerations

Future versions may support multiple repositories per Project.

Example:

```text
Project
    |
    +---- Frontend Repository
    +---- Backend Repository
    +---- Infrastructure Repository
```

For this reason, Atlas treats Project and Repository as separate concepts even though Version 0.1 supports a one-to-one relationship.

---

## Scan

A Scan represents a single execution of repository analysis.

A Scan is created whenever Atlas analyzes a Repository.

Each Scan produces:

- Extracted Facts

- Generated Summaries

- Historical Records

Multiple Scans may exist for the same Project.

Example:

```text
Project
   |
   +---- Scan #1
   +---- Scan #2
   +---- Scan #3
```

Atlas preserves scan history.

---

## Fact

A Fact is a structured piece of information extracted from a Repository.

Facts are objective observations discovered during analysis.

Examples:

- File exists

- Directory exists

- Dependency exists

- Documentation exists

Facts should be machine-readable and reproducible.

Examples:

```text
Directory: src/

File: README.md

Dependency: fastapi
```

Facts are the foundation of Atlas knowledge.

---

## Summary

A Summary is human-readable knowledge generated from Facts.

Summaries help users understand a project without inspecting raw data.

Examples:

- Project Overview

- Structure Summary

- Dependency Summary

Example:

```text
This project is a Python application built using FastAPI and SQLAlchemy.
```

Summaries are derived from Facts.

---

## Knowledge

Knowledge is the collective understanding Atlas builds about a Project.

Knowledge consists of:

- Facts

- Summaries

- Historical Scan Information

Knowledge is the primary output of the Atlas platform.

---

## Dependency

A Dependency is an external package, library, or component required by a Repository.

Examples:

- fastapi

- sqlalchemy

- react

- express

Dependencies are discovered during repository analysis.

---

## Directory

A Directory is a folder discovered during repository analysis.

Examples:

```text
src/
tests/
docs/
```

Directories are stored as Facts.

---

## File

A File is an individual file discovered during repository analysis.

Examples:

```text
README.md
requirements.txt
package.json
main.py
```

Files are stored as Facts.

---

## Repository Analysis

Repository Analysis is the process Atlas uses to understand a Repository.

Typical steps include:

1. Retrieve Repository

2. Scan Structure

3. Extract Facts

4. Store Facts

5. Generate Summaries

6. Store Knowledge

Repository Analysis does not modify repositories.

---

## Historical Scan Data

Historical Scan Data represents knowledge generated by previous Scans.

Atlas retains historical scan information to support:

- Project evolution tracking

- Change analysis

- Future comparison capabilities

Historical data is considered a core platform capability.

---

## Agent

An Agent is a role-based participant in the Atlas development workflow.

Agents are responsible for specific activities.

Examples:

- Architect Agent

- Developer Agent

- Reviewer Agent

- Project Manager Agent

Agents follow documented responsibilities and constraints.

---

## Task

A Task is a unit of work created from project requirements and milestones.

Tasks define:

- Objective

- Scope

- Requirements

- Acceptance Criteria

Tasks are executed by the Developer Agent and validated by the Reviewer Agent.

---

## Milestone

A Milestone is a collection of related Tasks that deliver a meaningful outcome.

Examples:

- Milestone 1: Project Management Foundation

- Milestone 2: Repository Scanning

- Milestone 3: Fact Extraction

Milestones define development progress.

---

## Architecture Decision Record (ADR)

An Architecture Decision Record (ADR) documents a significant architectural or product decision.

Each ADR contains:

- Decision

- Reasoning

- Consequences

- Status

Example:

```text
ADR-001

Decision:
Support a single repository in Version 0.1.

Reason:
Reduce MVP complexity.
```

ADRs preserve project knowledge and decision history.
